# KL2003 F2 k=3 Data Certificate Format Scoping v1

Date: 2026-07-18

Status: `K3_DATA_CERTIFICATE_FORMAT_SCOPED`

## Purpose

This note fixes the minimal data-certificate contract for the k=3
generator/verifier pilot and records the first high-k-parametric schema
decision.

It does not generate k=3 rows, does not solve an LP, and does not open Lean.
It specifies the artifact that a future untrusted generator may emit and that a
future trusted Lean verifier must independently check.

The certificate is data, not a proof.  The schema is intended to be
k-parametric: k=3 is the pilot instance, while the same shape should be
stress-tested before any k=9/k=11 attempt.

Schema version:

```text
KL2003_F2_HIGH_K_DATA_CERTIFICATE_FORMAT_v2
```

Nested-EL amendment version:

```text
KL2003_F2_HIGH_K_DATA_CERTIFICATE_FORMAT_v2_1
```

Compatibility policy:

- v2 remains valid for flat/simple row artifacts such as fixtures,
  k2-regression reemission, row25, and row22.
- v2.1 is required for nested EL rows such as source-faithful row28 V3.
- Historical v2 artifacts should not be rewritten merely because v2.1 exists.
- A validator must accept both v2 and v2.1, applying v2.1-only checks only
  when `metadata.schema_version = KL2003_F2_HIGH_K_DATA_CERTIFICATE_FORMAT_v2_1`.

## Artifact Layout

The future generator should emit a directory such as:

```text
outputs/KL2003_F2_K3_GENERATOR_VERIFIER_PILOT_v1/
```

The canonical external-audit artifact is:

```text
kl2003_k3_certificate.json
```

For fixtures only, the filename must make that status explicit:

```text
kl2003_k3_certificate.fixture.json
```

The future generator must also emit a deterministic Lean data file generated
from the JSON:

```text
KL2003K3CertificateData.lean
```

For fixtures only, the Lean file must also be clearly marked:

```text
KL2003K3CertificateDataFixture.lean
```

The manifest is:

```text
manifest_sha256.csv
```

Auxiliary diagnostics may exist.  The trust split is:

```text
JSON = canonical external-audit/provenance artifact
generated .lean = deterministic Lean-facing data artifact
Lean verifier = trusted checker over generated .lean data
```

Lean does not import JSON.  The verifier consumes generated Lean definitions.
The JSON remains the source of auditability and reproducibility.

## Manifest Contract

`manifest_sha256.csv` must hash every emitted artifact in the pilot output
directory, including `kl2003_k3_certificate.json`.

Required manifest columns:

```text
path
sha256
artifact_kind
generator_version
schema_version
created_at
source_commit
json_sha256
lean_data_sha256
json_to_lean_generator_sha256
notes
```

Meanings:

- `path`: relative path inside the output directory.
- `sha256`: SHA256 of the artifact bytes.
- `artifact_kind`: one of `certificate`, `diagnostic`, `summary`, `hook`,
  `manifest_input`, or `other`.
- `generator_version`: stable generator version string or generator git hash.
- `schema_version`: schema identifier, initially
  `KL2003_F2_HIGH_K_DATA_CERTIFICATE_FORMAT_v2`.
- `created_at`: UTC timestamp generated by the producer.
- `source_commit`: repository commit or `UNKNOWN_UNCOMMITTED` if the run is not
  attached to a clean commit.
- `json_sha256`: hash of the canonical JSON certificate.
- `lean_data_sha256`: hash of the generated Lean data file.
- `json_to_lean_generator_sha256`: hash of the deterministic JSON-to-Lean data
  generator.
- `notes`: free-form review note, not proof data.

The manifest itself should be included in downstream custody by hashing it from
the outside, for example in the run summary or a later audit package.

## JSON Top-Level Structure

`kl2003_k3_certificate.json` must contain at least:

```json
{
  "metadata": {},
  "tracked_classes": [],
  "rows": [],
  "inverse_words": [],
  "deletion_marks": [],
  "rational_certificate": {},
  "slacks": [],
  "verification_targets": []
}
```

No top-level field is trusted merely by appearing in the artifact.  The future
Lean verifier must either check the field directly or ignore it.

For v2.1 nested EL artifacts, the JSON must additionally contain:

```json
{
  "nodes": [],
  "nested_blocks": [],
  "post_deletion_edges": [],
  "sibling_disjointness_groups": [],
  "overlap_guards": [],
  "source_refs": [],
  "termination_policy": {}
}
```

The v2.1 additions are mandatory for nested EL and optional for v2 flat rows.
They are data, not proof.

```text
run_id

k
tracked_class_convention
pre_reduction_class_count
tracked_class_count
artifact_links
generator_version
created_at
```

Expected values for the first pilot:

`source_inputs` should include paths and hashes for the scoping document and

`artifact_links` should include:

```text
json_to_lean_generator_sha256
lean_import_policy
```

If the JSON cannot contain its own stable hash without self-reference, the hash

## `tracked_classes`

Each tracked class entry should contain:

```text
class_id
modulus
representative
pre_reduction_residues
source_rule
active
notes
```

All numeric values are strings, not JSON numbers.  For example:

```json
{
  "class_id": "k3_class_0",
  "modulus": "27",
  "representative": "5",
  "pre_reduction_residues": ["5", "14", "23"],
  "source_rule": "REQUIRES_SOURCE_RULE_ID",
  "active": true,
  "notes": "placeholder shape only"
}
```

The example above is illustrative only.  It is not k=3 generated data.

## `rows`

Each row must be sufficient to reconstruct the abstract row obligation.

Required fields per row:

```text
row_id
source_class
row_kind
children
deletion_policy
coefficient_terms
target_bound
slack_id
source_ref
normalization_ref
```

Field meanings:

- `row_id`: stable unique identifier.
- `source_class`: tracked class id of the caller/root.
- `row_kind`: for example `raw`, `EL`, `retarded`, `post_deletion`, or a more
  specific source-faithful tag.
- `children`: list of child/literal references, inline or by `child_id`.
- `deletion_policy`: id of the deletion rule used by this row.
- `coefficient_terms`: exact rational coefficients and endpoint references.
- `target_bound`: declared right-hand/target coefficient bound.
- `slack_id`: link to the row's slack record.
- `source_ref`: source line/table/rule reference.
- `normalization_ref`: local normalization note or generator rule id.

## Child / Literal Fields

Every child/literal appearing in a row must include:

```text
child_id
inverse_word
target_class
shift
window_policy
fiber_parent
deleted
reason
```

Recommended additional fields:

```text
row_id
literal_id
root_formula
forward_route
residue_side_conditions
coefficient_ref
depth
boundary_correction
```

Field meanings:

- `child_id`: stable unique id within the certificate.
- `inverse_word`: finite edge word in the agreed notation, for example
  `["even", "odd"]` or `["e", "o"]`.
- `target_class`: tracked class id after all source-faithful lifts/reductions.
- `shift`: exact symbolic or rational-shift descriptor; if numeric, it must be
  an exact rational string.
- `window_policy`: id of the floor/ceil/pad policy for this literal.
- `fiber_parent`: predecessor-of-entry parent used for disjointness/fiber
  proofs.
- `deleted`: boolean deletion marker.
- `reason`: source rule or local normalization reason for keeping/deleting the
  literal.

## `deletion_marks`

Deletion information may be duplicated inline in children for readability, but
the canonical deletion ledger is `deletion_marks`.

Required fields per mark:

```text
row_id
child_id
deleted
deletion_policy
reason
boundary_correction
source_ref
```

The verifier must reject a certificate where a row and its deletion ledger
disagree.

## v2.1 Nested EL Fields

The v2.1 amendment adds canonical fields needed by row28 V3 and later high-k
nested EL rows.

### `nodes`

Each tree node or generated literal node should contain:

```text
node_id
node_status
figure_node_id
parent_node_id
class
shift
branch_role
contributes_to_cardinality
source_ref
```

Allowed `node_status` values:

```text
counted
deleted
terminal
expanded
```

Deleted nodes must not contribute directly to cardinality.

### `nested_blocks`

Each block must contain:

```text
block_id
block_kind
children
```

Allowed `block_kind` values:

```text
min
sum
row
auxiliary
```

Block children may reference:

```text
block_ref
node_ref
component
```

Every `block_ref` must resolve to a declared `block_id`, and the block graph
must be acyclic.

For row28 V3:

```text
M1V3 second arm = phi25
M1V3 second arm != phi22
M2V3 is represented as a nested block
```

### `post_deletion_edges`

Each post-deletion edge must contain:

```text
edge_id
deleted_node_id
replacement_child_ids
reason
source_ref
```

`deleted_node_id` must refer to a node whose status is `deleted` or `expanded`.

### `sibling_disjointness_groups`

Each group must contain:

```text
group_id
parent_fiber
member_node_ids
reason
```

A `sum` block that claims disjointness must either refer to such a group or
carry an equivalent checked group id.

### `overlap_guards`

Each guard must contain:

```text
guard_id
guard_kind
scope
reason
```

The required guard for row28 is:

```text
guard_kind = no_parent_descendant_sum
```

The validator must reject a sum that includes both a node and one of its
descendants.

### `source_refs`

Nested EL artifacts must distinguish provenance roles:

```text
source_ref_id
source_kind
path
line_range
sha256
trust_role
```

Figure A1/root_paths references are regression oracles, not generator sources:

```text
trust_role = regression_oracle
```

### `termination_policy`

Nested EL artifacts must record the expansion termination policy:

```text
kind
status
source_ref
notes
```

The row28 gate currently requires representability of:

```text
kind = expand_until_deletion_saturation
status = hypothesis_untested
```

This records the policy as a pending hypothesis.  It does not prove termination
by saturation.

## `rational_certificate`

This section records exact arithmetic data for the pilot certificate.

Required fields:

```text
lambda_interval
coefficient_intervals
variables
row_lhs
row_rhs
solver_manifest
float_diagnostics
```

Rules:

- `lambda_interval` must use exact rational endpoint strings.
- `coefficient_intervals` must use exact rational endpoint strings.
- `variables` must be exact rationals or exact rational intervals.
- `row_lhs` and `row_rhs` must be reconstructible from row ids and coefficient
  refs.
- `solver_manifest` is evidence about how candidates were found, not proof.
- `float_diagnostics` may exist for debugging but cannot be consumed by Lean as
  evidence.

## `slacks`

Each slack entry must contain:

```text
slack_id
row_id
lhs
rhs
slack
strictly_positive
denominator_bits
numerator_bits
notes
```

`lhs`, `rhs`, and `slack` must be exact rational strings.

The verifier must prove:

```text
slack = rhs - lhs
0 < slack
```

or the equivalent inequality direction chosen by the row contract.

## `verification_targets`

This section declares exactly what the future verifier is expected to check.

Minimum target ids:

```text
row_well_formed
residue_compatibility
inverse_word_forward_route
deletion_rule_soundness
disjointness_or_fiber_partition
rational_slack_positive
abstract_row_discharge
manifest_matches_certificate
```

Each target should include:

```text
target_id
input_sections
expected_checker
status
notes
```

Initial status values should be data statuses, not proof claims:

```text
PENDING_LEAN_VERIFIER
PENDING_SCRIPT_VERIFIER
DIAGNOSTIC_ONLY
OUT_OF_SCOPE
```

## Exact Rational Form

Every numeric constant in the certificate must be encoded as a canonical string:

```text
"0"
"1"
"27/20"
"119/135"
"-3/10000"
```

Canonical rules:

- the string is either an integer or `num/den`;
- `num` and `den` are decimal integers with no leading zeros, except `"0"`;
- the fraction is reduced;
- the denominator is positive;
- the sign appears only on the numerator;
- integers are written as `"n"`, not `"n/1"`;
- zero is written only as `"0"`, not `"-0"` or `"0/den"`;
- JSON numbers are forbidden for mathematical constants.

Decimal strings are forbidden as final evidence:

```text
"1.35"        forbidden
1.35          forbidden
```

JSON numbers are forbidden for mathematical constants because parsers may
silently coerce them through floating representations.  Counts should also be
encoded as canonical integer strings for uniformity.

## Trust Rules

- The generator may emit incorrect data.
- The certificate may contain false rows, bad residues, bad deletion marks, or
  bad slacks.
- The Lean verifier must reject malformed or false data.
- The JSON artifact is not a proof.
- The generated Lean data file is not a proof by itself.
- The verifier is the proof boundary.
- Empirical hooks may accompany the artifact, but hooks are diagnostic and
  non-trusted.
- Solver logs and floats are provenance/debugging material only.
- v2.1 nested blocks remain untrusted data until the Lean verifier proves the
  corresponding well-formedness, route, deletion, disjointness, and slack facts.

## Anti-Wrapper Rules

The future Lean design must not wrap the JSON certificate or the generated Lean
data file as an axiom.

Forbidden shape:

```lean
axiom k3_certificate_valid : K3Certificate.Valid generatedCertificate
```

Required shape:

```text
1. Convert artifact data into concrete Lean definitions or generated data
   declarations.  The JSON-to-Lean conversion must be deterministic and
   hash-custodied.
2. Prove named well-formedness theorems.
3. Prove named residue/route/deletion/disjointness theorems.
4. Prove named rational slack positivity theorems.
5. Assemble a final data-validity theorem from those checked pieces.
```

Every mathematical obligation must have a theorem name.  The generator may make
the theorem file, but the kernel must check it.

Lean verifier policy:

```text
JSON is provenance.
Generated .lean is data.
Verifier theorems check the generated data.
```

## Acceptance Criteria For The Format

The format is acceptable for the k=3 pilot if it is:

- sufficient to reconstruct every generated row;
- sufficient to regenerate the Lean data file deterministically from JSON;
- sufficient to cross-check `json_sha256`, `lean_data_sha256`, and
  `json_to_lean_generator_sha256`;
- sufficient to verify every residue and class transfer;
- sufficient to replay each inverse word by forward routes;
- sufficient to check deletion marks and boundary corrections;
- sufficient to verify fiber/disjointness obligations or name the missing
  obligations precisely;
- sufficient to recompute every rational slack exactly;
- small enough for a k=3 pilot artifact to be reviewed by humans;
- regular enough to extrapolate to k=9/k=11 generator outputs;
- explicit about which fields are proof-relevant and which are diagnostics.

The format is not accepted if successful checking would require trusting the
generator's assertion that a row is valid.
